<project_specification mode="greenfield">
  <project_name>meridian</project_name>

  <overview>
    Meridian is a self-hosted, local-first control plane for AI agents that fuses
    Anthropic Managed Agents primitives (Session / Harness / Sandbox), OpenClaw's
    multi-channel Gateway, and hermes-agent's Skill Forge + Model Router into a
    single daemon. It is simultaneously a coding/dev agent (CLI + editor plugin)
    and a personal-assistant gateway (Telegram / Slack / Discord / webhook), with
    one durable Session reachable across every surface. SOTA differentiators:
    API-compatible with Anthropic beta SDK, capability-by-intersection sandboxing,
    deterministic replay with fixtures, mid-session checkpoints, self-improving
    Skill Forge producing agentskills.io-compatible skills, two-mode Anthropic
    provider (subscription OAuth + API key) for dev-loop economics, polyglot
    Python daemon + TypeScript UI bridged by OpenAPI codegen.
  </overview>

  <target_audience>
    P1 — Dev-loop user (coding agent): plans and executes work in a workspace,
    cares about tool safety, deterministic replay, multi-agent supervision, branch
    and PR workflows. Lives in CLI and editor plugin.

    P2 — On-the-go user (PA gateway): reaches the agent on Telegram or Slack —
    "what's my todo for today?", "summarize the project meeting". Cares about
    memory continuity, channel availability, voice.

    P3 — Platform/infra engineer: self-hosts Meridian on a team server. Cares
    about deployment, sandbox isolation, secret management, multi-user separation,
    audit logs, cost controls.

    P4 — Tool/skill author: publishes reusable tools (MCP-compatible or HTTP) and
    skills (agentskills.io-shaped). Cares about distribution, versioning,
    capability declarations.
  </target_audience>

      <primary_v1>SQLite WAL (single-daemon laptop install; zero external services)</primary_v1>
      <primary_v1_2>PostgreSQL 14+ supported tier (Repository pattern means zero application change)</primary_v1_2>
      <vectors_v1>sqlite-vec extension for vector embeddings</vectors_v1>
      <vectors_v1_2>pgvector for KB and MemoryStore vectors</vectors_v1_2>
      <full_text>SQLite FTS5 (BM25) for keyword retrieval</full_text>
      <event_log>NDJSON files (append-only, fsync configurable) per Session under $STORAGE_ROOT/events/&lt;YYYY&gt;/&lt;MM&gt;/&lt;DD&gt;/&lt;session_id&gt;.ndjson</event_log>

    </database>
    <communication>
      <api_style>REST mirroring Anthropic beta SDK paths under /v1, extensions under /v1/x</api_style>
      <streaming>Server-Sent Events (SSE) for server-to-client streams; resumable via Last-Event-ID; HTTP POST for client tool-result submission</streaming>
      <response_format>JSON with error envelope: { "error": { "code": "string", "message": "string", "details": {} } }</response_format>
      <ids>ULIDs throughout with typed prefixes (agent_, sess_, thr_, msg_, tc_, skill_, env_, mem_, vault_, usr_, chan_, file_, wh_)</ids>
      <content_blocks>Match Anthropic wire format: text / tool_use / tool_result / image / thinking</content_blocks>
    </communication>
    <infrastructure>
      <process_model>Single daemon (meridiand) hosts HTTP API, Session Service, Harness Pool, Gateway, Event Bus, Model Router, Environment Manager, Vault, KB, observability exporters</process_model>
      <skill_forge>Separate background worker process with its own rate-limited model budget</skill_forge>
      <workers>Sandboxed workers spawn per tool/hook call; local-backend workers are warm-pooled subprocesses; container backends call out to Docker / Modal / Daytona / Vercel / Singularity / SSH</workers>
      <observability>OpenTelemetry traces (one per session); Prometheus metrics on /metrics; JSON structured logs to stderr</observability>
      <config>Single YAML config at ~/.meridian/config.yml with secret_ref://vault/{id}/{key} indirection; hot-reload via SIGHUP and POST /v1/x/config/reload (validate-then-atomic-swap)</config>
      <codegen>FastAPI exports packages/schemas/openapi.yaml; openapi-typescript generates packages/sdk-ts; datamodel-code-generator generates packages/sdk-py; CI fails on drift</codegen>
      <ci_lint>uv run ruff + pyright + lint-imports (Python); pnpm biome + tsc --noEmit (TypeScript)</ci_lint>
    </infrastructure>
  </technology_stack>

  <prerequisites>
    - Python 3.11+ with uv installed (`curl -LsSf https://astral.sh/uv/install.sh | sh`)
    - Node.js 20+ and pnpm 9+ for the UI workspace
    - SQLite 3.38+ with FTS5 and the sqlite-vec extension available at runtime
    - Docker Desktop (for the docker environment backend; optional for laptop dev)
    - Optional: Claude Code CLI installed and `claude login` completed for the anthropic-oauth provider
    - Optional: API keys for Anthropic / OpenAI / OpenRouter stored in OS keychain (managed by Meridian Vault on first use)
    - Optional: Ollama running locally on :11434 for the local provider path
    - Environment variables: MERIDIAN_CONFIG (override config path), MERIDIAN_WORKSPACE (default workspace root); documented in .env.example
    - OS Keychain support (macOS Keychain / Windows Credential Manager / libsecret on Linux) for the default Vault backend
  </prerequisites>

  <key_interactions>

    <interaction number="1" name="Dev-loop: spawn a planner that fan-outs 3 workers">
      1. User runs: meridian run --agent planner-v1 "refactor the auth module into per-flow files"
      2. Planner Agent emits a tool_use for parallel_runs with 3 subtasks (capability subset: fs.read+fs.write on src/auth/**, exec.shell on lint/test commands only)
      3. Harness spawns 3 child Sessions, each with parent_session_id pointing at planner, each with declared cap subset (verified subset of parent grants)
      4. Children run in parallel, each emits its own event log; usage.delta events roll up to the parent
      5. As each child completes, child_session.completed events fire; planner reads results via thread.list_messages and synthesizes a final PR description
      6. User sees the multi-agent tree in the CLI TUI (or the Web UI's /sessions/:id page) with budget rollups per child
    </interaction>

    <interaction number="2" name="PA-gateway: Telegram → cross-channel followup via webhook">
      1. User DMs the Meridian Telegram bot: "What's on my calendar today?"
      2. Telegram driver receives update, posts to POST /v1/channels/{tg_chan_id}/inbound
      3. Gateway resolves remote_id → primary UserProfile via existing pairing; creates or wakes a Session attached to assistant Agent
      4. Harness wakes, calls calendar tool (HTTP backend, credential via in-harness proxy), composes reply
      5. Reply emitted to the Session, fans out via egress to attached Telegram channel
      6. User asks: "Tell me when my 2pm moves." → Agent registers a Cron + Webhook combination; on webhook fire, the Telegram driver posts the followup
    </interaction>

    <interaction number="3" name="Deterministic replay of last week's failed run">
      1. User opens /sessions/:id/replay in the Web UI for a failed session from 7 days ago
      2. UI fetches the recorded fixtures from $STORAGE_ROOT/fixtures/&lt;session_id&gt;.*.ndjson
      3. User clicks "Run replay" → POST /v1/x/sessions/{id}/replay with into_session_id=sess_new
      4. Harness runs with FakeModelAdapter + FakeSandboxAdapter; events stream live into the UI
      5. Divergence detected at event seq 47 (agent picked tool exec instead of the recorded read); replay aborts with structured error
      6. UI highlights the diverging event and links to the prompt change in the active Agent version that caused the divergence
    </interaction>

    <interaction number="4" name="Mid-session checkpoint and human handoff">
      1. A 4-hour migration session is at event seq 1240, phase waiting_for_tool, pending tool call to apply DB migration
      2. User clicks "Checkpoint" in the Web UI → POST /v1/x/sessions/{id}/checkpoint
      3. SessionCheckpoint written atomically; session phase transitions to paused
      4. User manually edits a config file and runs a manual smoke test
      5. User clicks "Resume" → POST /v1/x/sessions/{id}/resume; harness loads latest checkpoint, re-dispatches the pending tool call, continues
      6. Daemon SIGKILL between checkpoint and resume → next daemon start auto-resumes paused sessions on harness wake
    </interaction>

    <interaction number="5" name="Skill Forge proposes a reusable skill from yesterday's session">
      1. Background Forge worker watches terminated sessions; finds a cluster of 4 sessions performing similar "audit-then-rotate-secret" sequences
      2. Forge calls the Model Router to distill a general skill body in agentskills.io shape
      3. Forge stores as PROPOSAL with derived_from_session_ids and (optionally) A/B comparison metrics on test cases
      4. Primary user opens /skills/forge in the Web UI; reviews the trajectory provenance and the A/B summary
      5. User clicks Approve → POST /v1/x/skill_forge/proposals/{id}/approve; SkillVersion promoted to active with audit-logged human approval
      6. The Skill is now available to Agents that opt-in via explicit per-Agent activation
    </interaction>

    <interaction number="6" name="Anthropic provider mode switching mid-conversation">
      1. Session is using anthropic-oauth (subscription billing) for a worker role
      2. Worker needs to summarize a 250k-token document; Model Router rule fires: estimated_input_tokens.gt 100000 → anthropic-api:claude-opus-4-7
      3. ModelCall switches provider for this call only; usage.delta records the provider and dollar cost
      4. After summary returns, subsequent calls revert to anthropic-oauth per the role=worker rule
      5. Budget tracker shows hybrid usage in the per-Session report: $0.41 across 1 api_key call + 19 oauth calls (oauth = $0)
    </interaction>

    <interaction number="7" name="Capability denial returns to model, not orchestrator crash">
      1. Agent is granted fs.read[src/**] and exec.shell, no net.fetch
      2. Agent emits a tool_use for fetch_url with a github.com argument
      3. Sandbox.execute() runs the capability intersection check → required net.fetch[github.com] not satisfied
      4. Synthetic tool_result with is_error: true and code=capability_denied returned to the model
      5. Model reads the error and chooses a different approach (uses local files via fs.read or asks the user for help)
      6. Audit log records the denied capability request with agent_id and tool_call_id
    </interaction>

    <interaction number="8" name="Hot reload after rotating an Anthropic API key">
      1. Operator rotates the api_key in 1Password and pushes the new value into the OS keychain Vault
      2. Operator runs: meridian config reload (or sends SIGHUP)
      3. Daemon validates current config (no changes to YAML structure), atomic-swaps provider registry
      4. anthropic-api provider's resolve(auth) is lazy — next call to api.anthropic.com uses the new key
      5. In-flight model calls on the old key complete naturally; new calls hit the new key — zero downtime
      6. Audit event records the config.reloaded action; provider count unchanged
    </interaction>
